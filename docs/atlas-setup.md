# MongoDB Atlas Setup Runbook

This runbook reproduces the provisioning of the Mise MongoDB Atlas cluster (RECIPE-066). Local development uses the Docker container from `docker-compose.yml` by default; Atlas backs PR preview deploys (RECIPE-047) and the production deploy (RECIPE-046).

## Topology

| | Cluster | Database | DB User | Used by |
|---|---|---|---|---|
| Staging | `Cluster0` (M0) | `mise_staging` | `mise_staging_user` | PR preview deploys, optional local opt-in |
| Production | `Cluster0` (M0) | `mise_production` | `mise_production_user` | Production Vercel deploy |

Staging and production share one cluster (M0 free tier) but live in separate databases, each with its own scoped DB user. A misconfigured staging connection string therefore cannot read or write production data.

## Provisioning steps

### 1. Project and cluster

1. Sign in to [cloud.mongodb.com](https://cloud.mongodb.com).
2. Create a new project named `mise` under the `Trevor's Org` organization.
3. **Build a Database** â†’ **M0 (Free)**.
   - Provider: AWS
   - Region: `us-east-1` (matches Vercel's default `iad1` Functions region)
   - Cluster name: `Cluster0`
4. Skip the user-creation prompt that appears after the cluster spins up â€” DB users are created manually below so each is scoped to a single database.

### 2. Databases

On the cluster's **Browse Collections** page, click **+ Create database** twice:

| Database name | Initial collection |
|---|---|
| `mise_staging` | `_init` (placeholder; Payload creates real collections on first connect) |
| `mise_production` | `_init` |

The `_init` collections can be dropped after Payload populates the real collections.

### 3. Database users

**Database Access** â†’ **Add New Database User**, once per environment:

| Username | Privilege | Database | Password |
|---|---|---|---|
| `mise_staging_user` | `readWrite` (Specific Privileges) | `mise_staging` | autogenerated, copied to 1Password |
| `mise_production_user` | `readWrite` (Specific Privileges) | `mise_production` | autogenerated, copied to 1Password |

Both passwords are stored in 1Password under entries named **Mise â€” MongoDB Atlas (staging)** and **Mise â€” MongoDB Atlas (production)**.

### 4. Network access

**Network Access** â†’ **Add IP Address** â†’ `0.0.0.0/0`.

The allowlist is fully open because Vercel does not publish stable egress IPs on the Hobby/Pro plans. Authentication is via the per-database scoped users with strong autogenerated passwords. If the project ever moves to Vercel Enterprise (Secure Compute), tighten this to the published Vercel egress range.

### 5. Connection strings

Format:

```
mongodb+srv://<user>:<password>@cluster0.<id>.mongodb.net/<database>?retryWrites=true&w=majority&appName=Cluster0
```

Both URIs are stored in 1Password alongside their passwords, and are written into the encrypted `apps/web/.env.development` (staging) and `apps/web/.env.production` (production) via dotenvx:

```sh
pnpm dotenvx set MONGODB_URI '<staging-uri>' -f apps/web/.env.development
pnpm dotenvx set MONGODB_URI '<production-uri>' -f apps/web/.env.production
```

**The database segment is required.** Atlas's "Connect â†’ Drivers" UI gives you a URI ending in `.mongodb.net/?...` (no DB). Without a DB in the path, the driver targets `test`, where the scoped users have no permissions â€” connections succeed but every query fails with `user is not allowed to do action [find] on [test.<collection>]`.

## Pointing local dev at staging Atlas

Default local development uses the Docker container at `mongodb://localhost:27017/mise`. To exercise the staging Atlas database from a local checkout:

1. Get the staging URI from 1Password.
2. Replace `MONGODB_URI` in `.env.local` with the staging URI (`.env.local` overrides `.env.development`).
3. `pnpm dev`.

Production credentials must never be used from a local machine.
