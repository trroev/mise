# Google Sheets → Payload Migration Mapping

This document captures the mapping from the legacy "Trevor Recipe Database Savory" Google Sheets workbook to the Payload `recipes` collection (and its referenced collections), the transformations the import script will need to perform, and the data quality decisions that have been made up front.

It is the contract for the migration script ticket (RECIPE-038+). When the script lands, anything that drifts from this document should be reflected by an update here.

---

## 1. Source overview

- **Workbook:** `Trevor Recipe Database Savory.xlsx` (Google Sheets export — link in [issue #37](https://github.com/trroev/mise/issues/37)).
- **Shape:** 188 sheets total — **1 `Template` sheet (discard) + 187 recipes (one per tab)**.
- **Layout:** every recipe tab follows the same fixed grid (see §2). All recipes are **component recipes** (sauces, broths, glazes, purées, doughs, garnishes, spice blends) rather than plated dishes.
- **Bakers' percentages:** the right-hand columns of each tab compute bakers' % and true % from the base quantities. These are derived values and have no analogue in the Payload schema — they are discarded, not migrated.

### 1.1 Per-sheet grid

All recipe tabs share this layout (0-indexed rows / columns):

| Cell range | Content |
|---|---|
| `A1` | Recipe title (the human-readable name; canonical) |
| `A3` | Header row: "Base Total Formula" |
| `E3` | Header row: "Desired Total Formula" (3× scaled formula — discard) |
| `I3:K3` | Headers: `Ingredient`, `Bakers %`, `True %` (all derived — discard) |
| `A4:C20` | Up to 17 ingredient rows: name (col A), quantity (B), unit (C) |
| `E4:G20` | "Desired" 3× scaled mirror of A:C — discard (derivable) |
| `I4:K20` | Ingredient name repeated + bakers'/true % — discard (derivable) |
| `A21` / `E21` | Total weight rows — discard (derivable) |
| `A24` | Header: "Method/Notes" |
| `A25:A33` (variable) | Method steps, **one step per row** |
| `J24` / `J25` | "Base Yield" (always `1.0`) / "Desired Yield" (almost always `3.0`) — discard (these are scaling multipliers, not yield in human units) |
| `A34` (variable) | Source attribution — restaurant name + year (e.g. "Aubergine", "Jeninni, 2015"). Always preceded by a blank row. |

## 2. Field-by-field mapping

### 2.1 Recipes collection

| Source | Target (`recipes.<field>`) | Required? | Transformation |
|---|---|---|---|
| `A1` (recipe title cell) | `title` | ✅ | Trim. **Use `A1`, not the sheet name** — sheet names are limited to 31 chars by Excel and 38 sheets have a more readable title in `A1` (e.g. sheet `Daikon, braised` → title `Braised Daikon`). |
| Derived from `title` | `slug` | ✅ | Generated by Payload's `slugField()` hook. |
| — | `description` | optional | **No source.** Leave blank; backfill manually if/when desired. |
| — | `heroImage` | optional | **No source.** All recipes import without a hero image. |
| — | `cuisine` | optional | **No source.** Leave unset. Backfill manually. |
| Footer attribution at `A34` (e.g. "Aubergine", "Jeninni, 2015") | `author` | optional | **Decision:** map verbatim into `author`. The field is `text`, the existing admin-description even acknowledges it is provisional ("TODO: convert to a relationship to `users` once user-submitted recipes ship"). Provenance is real metadata and worth preserving. 8 recipes have no attribution — leave `author` blank. |
| — | `course` / `difficulty` / `dietaryTags` | optional | **No source.** Leave unset. These are component recipes, not plated dishes — `course` is largely meaningless for them anyway. |
| — | `prepTime` / `cookTime` | optional | **No source.** Leave unset. `totalTime` is auto-computed by the existing `computeTotalTime` hook. |
| Sum of ingredient `quantity` (col B) for rows in `g` | `yield.quantity` + `yield.unit` | optional | Set `yield = { quantity: <sum_grams>, unit: "g" }` when **all** ingredient rows for the recipe are weights (158/187 = ~85%). For mixed-unit recipes, leave `yield` unset (see §3.2). |
| `A4:C20` (one row per ingredient) | `ingredientGroups[0].ingredients[]` | ✅ (`minRows: 1`) | **Source is flat — wrap in a single synthetic group with no `groupLabel`.** See §2.3 for per-row mapping. |
| `A25:A33` (one step per row) | `instructionGroups[0].steps[]` | ✅ (`minRows: 1`) | **Source is flat — wrap in a single synthetic group with no `groupLabel`.** Each non-empty col-A cell after `A24` (and before the source-attribution footer) becomes one step with `description = <cell text>` and `timerMinutes` left unset. See §3.3 for the 5 recipes with no method. |
| — | `_status` / `publishedAt` | n/a | Import as **drafts** (`_status: "draft"`). Do not auto-publish — let the chef review each in the admin before flipping to published. |

### 2.2 Discarded source columns / rows

These contain no information that isn't derivable from other cells, so the script ignores them:

- **`E:G` "Desired Total Formula" block** — exact 3× of `A:C`.
- **`I:K` ingredient name + Bakers % + True %** — derivable.
- **`A21`/`E21` total-weight rows** — derivable.
- **`J24`/`J25` "Base Yield" / "Desired Yield"** — these are scaling multipliers, not yield in human units. Always `1.0` / `3.0` for 154/187 recipes; the variable values on the other 33 are also just scaling factors (e.g. `Desired = 0.5` means "half batch"). The real yield is the total ingredient weight.

### 2.3 Per-ingredient-row mapping

Each ingredient row in `A4:C20` becomes one entry under `ingredientGroups[0].ingredients[]`:

| Source cell | Target | Transformation |
|---|---|---|
| `A<n>` (e.g. `Onion, diced`) | `name` + `prepNote` | **Split on the first comma.** Left side becomes `name` (e.g. `Onion`), right side becomes `prepNote` (e.g. `diced`). See §3.1 for the heuristic and exceptions. |
| `B<n>` (numeric) | `quantity` | Coerce to number. **Empty cell → see §3.4** ("to taste" handling). |
| `C<n>` (text, e.g. `g`, `ea`, `tbsp`) | `unit` (relationship to `units`) | Normalise (case-fold, strip punctuation) and resolve to the `units` collection by abbreviation. See §2.4. |

### 2.4 Unit normalisation map

The full unit set observed across **1,123 ingredient rows**:

| Source value | Count | Target `units.abbreviation` | Notes |
|---|--:|---|---|
| `g` | 1069 | `g` | Direct. |
| `ea` | 34 | `ea` | Direct. |
| `drops` | 4 | — | **No SI mapping.** Treat as `ea` with a note? See §3.5. |
| `pt` | 4 | `pt` | Direct (Pint). |
| `"` | 3 | — | **Inches** (e.g. `Kombu | 3 | "`). No length unit in schema; convert to `ea` with `prepNote = "3-inch piece"` injected. See §3.5. |
| `sprig` | 2 | — | **Not seeded.** Either add a "Sprig" count unit during seed, or coerce to `ea` with prepNote. **Decision:** add `Sprig` to `UNIT_SEEDS` (count type). |
| `L` | 1 | `l` (Liter) | Case-fold. |
| `as needed` | 1 | — | "To taste" pattern — see §3.4. |
| `tsp` | 1 | `tsp` | Direct. |
| `tbsp` | 1 | `tbsp` | Direct. |
| `C` | 1 | `cup` | Likely "C" = cup (capital). Confirmed by recipe context — accept and remap. |
| `qt` | 1 | `qt` | Direct. |
| `pinch` | 1 | `pinch` | Direct. |

The script will keep an explicit normalisation table (case-folded source → target abbreviation) rather than infer.

### 2.5 Referenced collections

| Collection | Pre-import action |
|---|---|
| `units` | Run the existing `UNIT_SEEDS` seed first, **plus** add `Sprig` (count). The seed covers everything else used. |
| `cuisines` | No source data → no seeding required for migration. Manual entry post-import. |
| `tags` | No source data → no seeding required for migration. |
| `ingredients` | The `ingredients` collection is **referenced by `defaultUnit`** but is **not** referenced from `recipes.ingredientGroups[].ingredients[].name` (which is just text). The source spreadsheet has 514 unique ingredient strings post-prep-note-stripping that are likely far fewer canonical ingredients (~200–300 estimated). Decision: **do not pre-populate the `ingredients` collection from this migration.** It is a separate concern (RECIPE-NNN-tbd), and recipe ingredient lines store the ingredient name as text per the current schema. |

## 3. Data quality issues & decisions

Each issue below is tagged **[manual]** (clean before import), **[script]** (handle in the migration script), or **[skip]** (do not migrate this aspect).

### 3.1 Comma-embedded prep notes in ingredient names — **[script]**

604 of 1,123 ingredient rows (54%) contain a comma in the name. The pattern is consistent enough to script, but the comma is overloaded:

- **Prep note** (most common): `Onion, diced`, `Tomato, peeled and seeded`, `Chicken, skin off, chopped`.
- **Variety / specifier**: `Salt, sea`, `Vinegar, balsamic`, `Flour, AP`, `Bay leaf, fresh`, `Gelatin, silver leaf`, `Cream, heavy`.
- **Format**: `Cucumber Sheet, frozen`, `Huckleberry, sous vide`.

**Decision:** the script splits on the **first comma** and writes:
- `name` = left side, trimmed
- `prepNote` = right side, trimmed

This is imperfect for varietal commas (`Salt, sea` → `name: Salt, prepNote: sea` rather than `Sea salt`) but is consistent and reversible. The chef can fix individual entries via the admin UI after import; a one-time post-import cleanup is acceptable since none of these are referenced as foreign keys.

### 3.2 Mixed units within a single recipe — **[script]**

| Distinct units in a recipe | Recipes |
|---|--:|
| 1 (typically `g`) | 158 |
| 2 | 25 |
| 3 | 2 |
| 4 | 1 |
| 5 | 1 |

Common mixes: `g` + `ea` (e.g. bay leaves), `g` + `L` (large stocks). The schema supports per-ingredient units, so this is not a problem for ingredient rows.

**Implication for `yield`:** `yield.unit` is a single text field, so for the 29 recipes with mixed units the script leaves `yield` unset rather than picking arbitrarily.

### 3.3 Recipes with no method/instructions — **[script]**

5 recipes have a `Method/Notes` header but no step rows underneath:

- `Fumet (needs rev.)`
- `Piperade, Sauce (needs Rev.)`
- `Potato Cauliflower Puree`
- `Preserved Lemon Citronette`
- `Special Sauce`

`recipes.instructionGroups[].steps` has `minRows: 1`, so these would fail validation as-is. **Decision:** the script inserts a single placeholder step with `description: "TBD"` for any recipe that has zero method rows. Combined with the import-as-draft policy (§2.1), the chef can fill these in via the admin before publishing.

### 3.4 Ingredients with blank quantity ("to taste" pattern) — **[script]**

35 ingredients (in 28 recipes) have a non-empty name but blank quantity cell (e.g. `Salt`, `Lime juice`, `Xanthan gum`, `Gelatin, silver leaf | as needed`). Common-sense reading: "to taste" / "as needed".

`recipes.ingredientGroups[].ingredients[].quantity` is `required: true` and `min: 0`. Two options:

1. Set `quantity = 0` and append `to taste` (or merge with the existing `as needed` unit value) to `prepNote`. **[chosen]**
2. Drop the ingredient row from the import.

**Decision:** option 1. Setting quantity to 0 and surfacing intent via `prepNote` preserves the chef's intent and is reversible.

### 3.5 Unmappable units (`drops`, `"`, `sprig`) — **[script]** + **[seed]**

- `sprig` (×2) → seed a new `Sprig` count unit and link directly.
- `drops` (×4) → import as `ea` with `prepNote` prefixed `"<n> drops, "`. Alternatively, a `Drop` count unit could be seeded; not done here because frequency is too low to justify, and "drops" is by convention an instruction-side modifier.
- `"` / inches (×3, all `Kombu` lengths) → import as `ea` with `prepNote` prefixed `"<n>-inch piece, "`.

### 3.6 Sheet-name truncation (Excel 31-char limit) — **[script]**

5 sheets are name-truncated by Excel's tab-name limit:

- `Apicius Spice Blend (needs rev.` → A1: `Apicius Spice Blend`
- `CarrotBrown Butter Whey Emulsio` → A1: `Carrot/Brown Butter/Whey Emulsion`
- `Orange Vanilla Gelee (needs rev` → A1: `Orange Vanilla Gelee`
- `Pistachio Pan de Genes (needs r` → A1: `Pistachio Pan de Genes`
- `Squab Sauce with Plum (needs Re` → A1: `Squab Sauce with Plum`

**Decision:** the script uses `A1` as the canonical title. The sheet name is only used to address the tab during reading.

### 3.7 "(needs rev.)" recipes — **[script]**

12 recipes have a `(needs rev.)` (or `(needs Rev.)`) marker in their tab name. These are recipes the chef has flagged for review.

**Decision:** import all recipes as `_status: "draft"` regardless. The 12 flagged ones are also dropped into the description's first line as `**Needs revision.**` so they surface in the admin. Strip the `(needs rev.)` suffix from the title before import.

### 3.8 Title casing & word order — **[script]**

38 recipes have an A1 title with different word order to the sheet name (`Daikon, braised` → `Braised Daikon`, `Banana Puree, Roasted` → `Roasted Banana Puree`). The A1 form is the canonical, user-facing title — use it.

### 3.9 Source attribution footer — **[script]**

179 of 187 recipes (96%) have a footer cell with restaurant + year (top values: `Aubergine` ×159, `Jeninni, 2015` ×6, `Reserve Wine & Food, 2014` ×3, plus 9 less-frequent variants). Map → `recipes.author` verbatim. 8 recipes with no attribution leave `author` blank.

### 3.10 Things explicitly skipped — **[skip]**

- `description`, `heroImage`, `cuisine`, `course`, `difficulty`, `dietaryTags`, `prepTime`, `cookTime` — no source. Manual backfill in the admin if/when the chef wants them.
- Bakers' % / true % columns — recomputable.
- "Desired" 3× formula — recomputable.

## 4. Proposed import order

> Subject to revision when the script ticket (RECIPE-038+) is scoped — recorded here so the script author starts from a known baseline.

1. **Seed** `units` via existing `UNIT_SEEDS` **plus** a new `Sprig` (count) unit.
2. **Read** the workbook tab-by-tab; build an in-memory list of recipe DTOs applying every transformation in §2 and §3.
3. **Validate** the full batch against the Payload `recipes` schema before any writes — fail fast on the first invalid recipe with a row-level error message.
4. **Insert** all recipes as drafts (`_status: "draft"`). Take advantage of Payload's local API for performance; no need to go through HTTP.
5. **Hand off** to the chef for review and per-recipe publishing in the admin.

`cuisines`, `tags`, and `ingredients` are not seeded by this migration.

## 5. Post-migration cleanup

Once the migration has been run successfully against production and the chef has confirmed the results, run the bulk-publish script and then remove the one-shot tooling:

```sh
# Flip every imported draft to "published" (uses Payload local API so the
# stampPublishedAt hook stamps each first-time publish):
dotenvx run \
  -f apps/web/.env.development.local \
  -f apps/web/.env.development \
  -- pnpm tsx scripts/publish-all-recipes.ts --env production --write

# Then clean up:
pnpm remove -w xlsx zod
rm scripts/migrate-from-sheets.ts scripts/publish-all-recipes.ts
```

`UNIT_SEEDS` and `scripts/seed-units.ts` stay — they are part of the regular environment bootstrap, not migration-only.
